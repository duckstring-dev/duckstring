# Duckstring

**Working with the author:** ask questions in **prose**, not the multiple-choice question UI — the author prefers a plain conversational back-and-forth.

A packaging standard for data transforms. Each transform is a versioned **Pond** (Python package) that declares its parent Ponds in `pond.toml`. The pipeline is implicit in the package graph. **Ripples** are the execution units within a Pond. The **Catchment** (FastAPI server) is the reference runtime — a convenience, not the product.

`docs/docs/theory.md` is the **authoritative orchestration spec** — its "Pond State Variables" pseudocode is the exact state machine. `playground/src/lib/orchestration.ts` is a well-tested TypeScript *simulation* (the standalone playground); the Python engine is a faithful, behaviour-for-behaviour port of it. The rest of `docs/` is the Docusaurus documentation site (→ docs.duckstring.com), written against the real CLI/API surface — update it when that surface changes.

## Brand & Positioning

(The old `brand/` directory was removed from the repo — internal strategy doesn't belong in OSS. These are the settled rules.)

- **Never describe Duckstring as an orchestration framework.** That positions it against Airflow/Prefect/Dagster. The differentiation is the package model, not the execution model. **Never mention competitors by name in copy or docs.**
- **The Catchment is not the product.** Don't lead with it in copy or docs introductions. It's the batteries-included runtime for teams that want the full stack.
- **Target audience**: data engineers who have hit the coordination and ownership walls of large transform pipelines — specifically those who've adopted or considered a mesh pattern and found that breaking changes still require organisation-wide coordination. They've reasoned their way to needing versioned package boundaries; they just don't have SemVer or concurrent version execution yet. Mesh-pattern users are the warmest first audience.
- **Tagline**: "There is no DAG." — the DAG exists but is implicit in the package graph. You don't build or govern it. **Package description** (pyproject/PyPI): "Build data pipelines the way you build software: version each transform, declare its dependencies, and Duckstring resolves the execution DAG automatically."
- **Name the gaps honestly, but frame scope right** — Duckstring runs on a single Catchment's compute (local or remote); state that plainly, but it is **not** a row-count ceiling. DuckDB works through large data progressively and Trickle processes only the per-run delta, so "single-machine" ≠ small data — never put a fixed row cap (the old "~50M rows" line) in copy. The only honest non-claim is multi-node/cluster compute, and don't frame *that* as a deficiency: most pipelines genuinely don't need distributed compute — the world has been upsold into it. Make the big, distributed jobs the exception, not the rule.
- The README is deliberately short (positioning → concepts → one golden-path quickstart → trigger table → docs links); examples live in `docs/`, not the README — duplicated examples drift, and the PyPI copy of a stale README is immutable per release. The README's opening paragraphs are the author's own wording — don't rewrite them.

## Current state (2026-06)

The freshness/push-token runtime is **implemented and tested** (the old generation/watermark/demand model and TypeScript-simulation-only era are gone from the backend). The backend + CLI are complete, and the **web UI is built** (read-mostly Next.js, polls the Catchment — see Web UI below). The **playground was extracted** to a standalone `playground/` (in-memory sim). **Fault tolerance is now implemented end-to-end** (immediate + on-change retries, failed/killed/blocked states, dead/silent-Duck detection, error+traceback surfacing — see Fault tolerance). The old `docs/guide/` design docs were removed; `docs/` is now a fully-written Docusaurus site (`docs/docs/theory.md` carried over — its Fault-Tolerance section IS current). **Puddles (local pre-deploy testing) are implemented** — see Puddles below.

**Concurrent major versions are implemented** (v0.2.0): the runtime Pond identity is the **pond key `"{name}@{major}"`** (`keys.py`) — each deployed major line is an independent live Pond with its own engine node, Duck, and storage (`ponds/{name}/m{major}/{registry.duckdb,data/,pond.db}`). Sinks wire to the Source major their `[sources]` pin selects; `Pond.read_table` resolves it via `source_majors` (computed by the Duck from the deployed `pond.toml`; absent for puddle runs → flat layout). All pond-targeting routes take optional `major`/`version` query params — `major` picks the line (default: highest deployed), `version` must be that line's *selected* artifact (422 otherwise) — resolved by `Driver.resolve`; the CLI `--major`/`-m`, `--version`/`-v` pass through everywhere; `/api/status` entries carry `id`/`name`/`major` and `edges` use ids. `min_version` from `[sources]` is still stored-but-unenforced. **Auth is two-model**: platform auth (recommended for hosted — the platform gates requests; the UI rides its session cookies; Ducks dial localhost inside the sandbox; the CLI attaches per-catchment custom headers via `catchment connect --header 'Name: value'`, stored in config and merged by `config.auth_headers`) and built-in API keys for bare self-hosting. The built-in keys are a **total-ordered access ladder** `read ⊂ demand ⊂ full` (`catchment/auth.py`): each route declares a minimum via `dependencies=[auth.read|demand|full]` (read=status/runs/data/draw/view, demand=tap/wave/pulse/tide/untrigger + duct-connect, full=deploy/control/windows/ducts/rotate); `audit_routes` runs at `create_app` and **fails closed** (an unclassified `/api` route — bar `/api/health` and the duck channel — raises at boot). A request's level = the matched key's; too-low = `403`, missing/invalid = `401`. The three keys are stored as **sha256 hashes** in `catchment_key` (migration `007`; `auth.generate` mints them, printed once); `init --generate-key` mints the ladder + stores the **full** key in the registration, `init --key`/`DUCKSTRING_API_KEY` is a single full key (legacy/bare), neither = open (`get_principal` → full). **Reroll** without recreating: `POST /api/catchment/keys/rotate` (full-gated) / `duckstring catchment rotate-keys [--level]` (updates the stored full key). The **Duck channel** (`/api/duck/*`, `auth.duck`) uses a **separate internal token** (`auth.ensure_duck_token`, persisted in `catchment_meta` so a Duck surviving a Catchment restart keeps authenticating; passed `--token=` joined so a leading `-` isn't read as a flag) — decoupled from the user keys, so rotating them never disrupts running Ducks and no subprocess carries a full key. The web UI prompts for a key on 401 and keeps it in localStorage. `_http.get/post` take `auth=cfg` (the registration dict). config.toml is chmod 0600. **The static UI is subpath-hostable** (e.g. Posit Connect `/content/{guid}/`): `assetPrefix: "./"` + a runtime-derived API base in `frontend/src/lib/api.ts`; `tests/test_static.py` + the release workflow guard against absolute asset paths. **Platform hosting is first-class**: `duckstring.catchment.asgi` is the env-configured entry (`DUCKSTRING_ROOT`, default `./.duckstring` — survives restarts but not redeploys of the bundle; run exactly ONE app process). **State download**: `duckstring catchment download [--path DIR=./.duckstring]` pulls the whole root via `GET /api/catchment/archive` (streamed uncompressed tar; SQLite snapshotted via the backup API, `-wal/-shm` skipped; DuckDB registries copied as-is — download while quiescent) after a size confirmation from `GET /api/catchment/usage`; the default path drops into a deploy bundle so state survives a platform redeploy (docs: running-a-catchment "Surviving a redeploy"). The Duck dial-back address is no longer assumed: `create_app(root, base_url=...)` (the CLI passes its bind address — this fixed `catchment init --port` spawning Ducks at :7474), env `DUCKSTRING_CATCHMENT_URL`, or None → `SubprocessLauncher` defers spawns (pending keys report `is_running` so liveness won't fail them) until a middleware learns the bound address from the first request's ASGI scope (`tests/test_platform.py`). **Release automation**: `release.yml` builds the frontend + dists on a `v*` tag and publishes via PyPI Trusted Publishing (the `pypi` GitHub environment + PyPI publisher must be configured once).

## Runtime architecture (two-tier: Catchment + Ducks)

- **The Catchment owns pull.** It runs the **full** engine (`engine/catchment.py`: Ponds *and* Ripples, pull + push), holds triggers/windows, and decides Pond Runs. Modelling ripples is required — the Tap-3/1 result and the bottleneck cadence come from *ripple-level* pull. `start_pond_run` records a `BeginRun(pond, F)` on `state.pending_begin_runs`; `Driver` drains these and dispatches them.
- **Each executing Pond runs a "Duck"** (`duck/`, one subprocess per pond key `name@major`, `SubprocessLauncher`). Given `begin_run(F)` it pushes every Ripple to `F` (push-only, `engine/worker.py`), executes ripple functions, and reports `ripple`/`run_completed` events. It is spawned on the first run, killed when the Pond is idle (kept warm while a standing trigger is active), and **survives Catchment downtime** (finishes in-flight runs from its ledger + engine, buffers events, replays idempotently on reconnect).
- **No cap** on concurrent Pond Runs — completions clock the pull cascade; that is the flow control.
- **Transport**: Duck→Catchment is REST POST (`/api/duck/{name}/{major}/events`); Catchment→Duck is a short-poll the Duck holds (`/api/duck/{name}/{major}/jobs`). The Duck always dials back, so the same code works local and (future) remote — remote is just a different launcher. `DUCKSTRING_CATCHMENT_URL` tells Ducks where to dial; `DUCKSTRING_DISABLE_DUCKS=1` swaps in `NoopLauncher` (tests exercise the engine + persistence without spawning processes).
- Cross-Pond data: each major line writes its tables to `ponds/{name}/m{major}/data/{table}.parquet` (atomic tmp+replace); sinks read the parquet of the Source major they pin. Per-line DuckDB registry at `ponds/{name}/m{major}/registry.duckdb`.

## Triggers & control (CLI → `/api/ponds/{name}/…` → Driver)

(Routes still live at `/api/ponds/{name}/…` and accept *any* deployed Pond, not only Outlets.)

**Triggers** (`cli/trigger.py` — demand signals):
- **tap** (one pull), **wave** (standing pull), **pulse** (push `now`, propagates upstream), **tide** (standing push; a **staleness bound** like `30s`/`1d`, not cron).
- **remove** — drop the standing Wave/Tide trigger (existing work drains).

**Control** (`cli/control.py` — a Pond's execution & health; see Fault tolerance):
- **wake** (`engine.wake_pond`) — a one-shot **non-propagating** pull: runs once when Sources are already fresher (`sourceF > startF`), without soliciting them. Clears failure/kill.
- **force** (`engine.force_pond`) — recompute now at the *current* freshness even with no upstream change (resets the Pond's + Ripples' `endF` so they re-run); does **not** advance freshness, so it does not propagate downstream. Clears failure/kill.
- **refresh** (`engine.refresh_pond`, `PondState.refresh_pending`) — flag the Pond so its **next** run is a cold **wipe-and-rebuild**: the Duck drops the registry (`executor.wipe`) and runs with `previous_f=NEVER` → full Source reads → a Trickle re-bootstraps (changelog re-seeded with the initial state at `floor=run.f`; any prior base/changelog dropped), raising the floor so downstream coverage-misses and reloads. **Lazy** (changes *how* the next run computes, not *when*), so it propagates honestly at the next genuine freshness. `--clear` un-flags. See `plans/refresh.md`.
- **repair** (`engine.repair_pond` = force+refresh; `Driver.repair` + `_advance_repair`) — force-rebuild a **connected** set of Ponds *now*, Driver-sequenced in topological order (each node released once its in-scope parents finish, so it reads their rebuilt output). Steps out of the demand model; the scope is marked `repairing` (`PondState.repairing`, blocks normal demand via `can_start_pond`; `derive_blocked` treats a repairing Source as blocking downstream). Connectivity is validated (the relaxed "connected through the selection" rule — `driver._connectivity_gap`); `--downstream`/the UI canvas-select extend the scope. A repaired Pond's *floor* only advances where freshness genuinely does (an inlet); the data rebuilds regardless.
- **sleep** (`engine.sleep_pond`, the old `stop`) — clear push+pull and Ripples' pull, KEEP Ripple push so started runs complete; cancels the standing trigger; `--upstream` reaches ancestors.
- **kill** (`engine.kill_pond`) — terminate the Duck process and park the Pond **killed** (terminal, supersedes retries) until a wake/force/clear.
- **clear** (`engine.clear_pond`) — reset a failed/killed Pond (no run); abandons the halted Run's phantom (`start_f → end_f`) so liveness won't re-fail it, and unblocks downstream.
- **failure-budget** — show / set the live retry budgets (`--immediate`, `--on-change`).
- One-shot commands (tap/pulse/wake/force) open the live status until the target settles (idle/failed/killed/blocked); standing ones stay open.

## Windows (batch availability on Inlets)

RFC-5545-flavoured recurrence (no cron anywhere — `croniter` is gone). `engine.core.Window(start_anchor, duration, freq_unit ∈ {SECOND,MINUTE,HOUR,DAY,WEEK}, freq_interval, valid_days ⊆ {MON..SUN}|None, until)`. Occurrences are `start_anchor + k·delta` filtered by valid_days/until; a Pond is "fresh until" the active window's end, with `D` = window duration. `Window.active_end`/`next_boundary` are O(1) (used per-tick in `pond_source_f`/`next_wake`); `Window.occurrences` (bounded) is used only for add-time overlap validation. Managed via `duckstring trigger window {pond} add|list|remove` (`cli/window.py`); operational config (CLI/API, survives redeploys), not declared in `pond.toml`. `add` requires only `--name`/`--every` (`--start` defaults to 00:00 today; `--duration` defaults to `--every` = back-to-back).

## Puddles (local pre-deploy testing — `local/`, `cli/puddle.py`)

A **Puddle** is a code-defined snapshot of a Source table, for testing a Pond before deployment. Definitions in `src/puddles.py` (untyped `@puddle("source.table")` or whole-source `@puddle("source")`; handle `p` has `con`/`path`/`write_table`/`write_path`/`catchment()` — see `core.py`). `duckstring pond hydrate` imports them (decorator side-effect, like `@ripple`) and materialises `puddles/ponds/{source}/data/{table}.parquet` — the catchment-root layout, so `Pond.read_table`'s foreign branch works unchanged with `root=puddles/`. Missing definitions skip with a warning (`--from-catchment` fills from the Catchment). `duckstring pond run [--ripple X] [--fresh]` is **one local Pond Run** (sequential topo order, no engine/freshness/Ducks): full runs reset `puddles/out/` (registry + exported parquet); a **self-puddle** (`puddles/ponds/{this_pond}/`) is copied in as the seed first, making incremental reruns idempotent. `duckstring puddle ls|show|query` inspects `./puddles` via in-memory DuckDB views (`"{pond}"."{table}"`; output overrides a same-named self-puddle). Entrypoints are declarable in `pond.toml` (`[pond] ripples`/`puddles`, defaults `src/pond.py`/`src/puddles.py`) and honoured by deploy + the Duck executor via `core.import_pond_module`. Tests: `tests/test_puddle.py`; demo `sales` carries a worked `src/puddles.py`.

## Egress (Spouts — `egress/`, `cli/spout.py`; see `plans/egress.md`)

The OSS "last mile": getting a Pond's output **out** to the systems a team already runs (object store, Postgres), pluggably. **Built so far: the construct + the `file://` snapshot driver, end-to-end.** A **Spout** is a Pond's egress binding `(pond, major, table|*, destination, mode, schedule)` — **operational config** (CLI/API, persisted in `pond_spout`, survives redeploys), like windows, **not** in `pond.toml` (destinations/credentials are environment-specific). The **destination** is a URI whose scheme picks the egress driver (`file`/`s3`/`gs`/`postgres`); **credentials are `${env:NAME}` or `${secret:NAME}` references** resolved only at egress time (`egress/credentials.py` `resolve`; **env-var-first** — `${env:}` reads the Catchment's process environment, `${secret:}` reads the write-only catchment secret store, see Secrets). `egress/destination.py` validates the scheme + credential-reference *syntax* + mode (`auto`/`full`/`append`) without resolving. CRUD is `Driver.add_spout`/`list_spouts`/`remove_spout`/`resync_spout` → `/api/ponds/{name}/spouts` (+ `/resync`, full-gated) → `duckstring spout add|ls|rm|resync {pond}`; a Spout name defaults to the table (scheme for all-tables), `-2`/`-3` on collision.

**The seam** (`egress/base.py`): `EgressDriver` Protocol (`capabilities`/`ensure`/`write_full`/`apply_delta`/`test_connection`) + `Capabilities(supports_delta, supports_delete, transactional)` + `get_egress(uri)` (scheme registry; a *known* scheme with no driver yet raises "not implemented yet" → the worker parks the Spout). **`test_connection(con)`** (the UI add-form *Test* button) probes connectivity/credentials **writing no data** — file:// write+deletes a probe, postgres ATTACHes + `SELECT 1`, s3/gs configures httpfs + lists the prefix — raising a **sanitised** error (never a credential); surfaced at `POST /api/ponds/{name}/spouts/test` (full-gated) → `{ok}`/`{ok:false,error}` (a connection problem is a 200 result, not a 5xx). **The object-store driver** (`egress/object_store.py`): snapshot `write_full` → `{prefix}/{table}.parquet`, `supports_delta=False` so the worker always full-writes. **`file://`** writes locally (atomic tmp+replace); **`s3://`/`gs://`** write via DuckDB `httpfs` + the secret manager (credentials from the URI query `?key_id=${env:..}&secret=${env:..}&region=..`, resolved at egress time — or the AWS credential chain for `s3://` with no key, HMAC required for `gs://`; the secret-`CREATE` error is masked so it can't echo a credential; the target URI carries no query so an error can't leak one either). **The Postgres CDC driver** (`egress/postgres.py`, `postgres://`, `capabilities={delta,delete,transactional}`): the flagship incremental sink — a merge Trickle's changelog **is** a CDC stream. Transport = the DuckDB `postgres` extension (`ATTACH` + plain DuckDB SQL against attached tables; no SQLAlchemy/psycopg). **apply_delta = delete-then-insert in one transaction** (delete the changed∪removed keys, re-insert the present rows — same net effect as `INSERT … ON CONFLICT` but portable). The **watermark lives in the destination** (`_duckstring_egress`, set in the same txn) → **exactly-once** across crashes. The **transactional-PK gate** (only a merge Trickle has a pk) is enforced at `add_spout` (`Driver._assert_transactional_pk`, on a published table) and again at egress.

**A Spout is a real Pond — the egress dual of a Pond Draw** (migration `012`). It has its own identity rows (`_create_spout` mirrors `_create_draw`: `pond_name` kind='outlet' named `{source}#{spout}`, a synthetic `pond_version` + one `'egress'` ripple, `pond.is_spout=1`), wired to its source via `pond_to_pond`, and runs as an engine node with a **standing Wake** (`PondState.standing_wake` + `Pond.is_spout`; engine `tick` re-arms a *non-propagating* pull whenever idle). It delivers when `sourceF > deliveredF`, **never solicits the source** (a Wake, not a Wave) and, being terminal, **never blocks anything**. The **egress worker is "the Spout's Duck":** the engine dispatches its `BeginRun` to `_pending_egress` (not a Duck — the `is_draw` branch in `_dispatch_begin_run` is the template; liveness skips it), the worker drains `take_spout_jobs`, delivers, and reports via `complete_spout_run`/`fail_spout_run` — **the same `pond_run`/`ripple_run` path a Pond uses**, so a Spout gets real run history + tracebacks + `/api/runs` for free (the old clone is gone). The **Control set applies, the Demand set does not**: `Driver.spout_wake/force/sleep/kill/clear` reuse the engine `clear_pond`/`kill_pond` + the persisted `armed` flag (Sleep/Kill disarm, Wake/Force re-arm, Force re-delivers from scratch) → `POST /api/ponds/{name}/spouts/{spout}/{action}` (full-gated) → `duckstring spout wake|force|sleep|kill|clear|resync`. A Spout rides `ponds[]` with `is_spout` (and its source→spout edge in `edges`), dashed in the UI like a Draw. `pond_spout` (re-keyed to the Spout's pond_id) holds just `table_name`/`destination`/`mode`/`armed`; fault/retry state is the node's `pond_state`/`pond_retry`. **Windows throttle a Spout's delivery** with the **same mechanics as a windowed Inlet**: `pond_source_f` returns the active window's end for a windowed spout, so the Wake fires once per window (holding in a gap / until the source publishes), the run is stamped with the window end (the throttle clock), and the worker ships the source's data at the source's real freshness (the job's `source_f`; the in-destination CDC watermark rides it, exactly-once). Because a Spout is a real pond, windows **reuse the existing `pond_window` CRUD verbatim** (`trigger window {source}#{spout}` / the UI `WindowEditor` on the spout key — zero new surface; `next_wake` also wakes at a windowed spout's boundaries). The old demand-aware **`--every`** schedule is **dropped** (a staleness bound would solicit the source, contradicting the passive Wake — keep the *source* fresh with a Tide/Wave, throttle the *spout* with a window).

**The worker** (`catchment/egress_worker.py`): an async loop in the Catchment process (like the poller), woken on run-completion/control (`Driver._signal_egress`), self-healing on a 5 s tick. It drains `Driver.take_spout_jobs` (the runs the engine dispatched) and for each: a **transactional, delta-capable** driver reads the changelog delta over `(in-dest watermark, f]` (`trickle_io.read_delta`) and `apply_delta`s — falling back to `write_full` on a full read (bootstrap / coverage-miss / changed overwrite source); others snapshot. On success → `complete_spout_run`, on failure → `fail_spout_run(... traceback)` — never the source. **The dashed-node frontend is built** (`PondNode` renders a Spout dashed + `[SPOUT]`; the Sidebar's `SpoutPanel` is its Control set + delivery-throttle `WindowEditor`; a source Pond's `SpoutEditor` lists + adds Spouts, credentials as *names* — env-var or stored-secret (a datalist of stored secret names), assembled into `${env:NAME}`/`${secret:NAME}`, a value never crosses the wire; the add form has a **Test** button (`POST .../spouts/test`, result tagged with its destination so an edit clears it); the secret store itself is managed from `SecretsMenu`, the **Secrets** button under Collapse-all (top-right `ControlsPanel`, full only) — see Secrets). The access badge is three capabilities (`Manage | Demand | Read`, each a green ✓ when granted / grey – when not; labels stay white). **Still to build:** the incremental object-store path (per-run parts / Iceberg-in-bucket). **Real-backend write e2es (MinIO/moto S3, containerised Postgres) are the CI follow-up** — locally the Postgres logic is tested against a DuckDB-attached destination (the same SQL path). Full design in `plans/egress.md`. Tests: `tests/test_spout.py`, `tests/test_egress_credentials.py`, `tests/test_egress_file.py` (+ a real-Duck e2e), `tests/test_engine.py` (the standing-Wake primitive + windowed-spout firing).

## Secrets (`catchment/secrets.py`, `routes/secrets.py`, `cli/secret.py`)

A **write-only** catchment-wide secret store for egress credentials referenced as `${secret:NAME}`. `SecretStore(root)` persists a plaintext `secrets.json` at the catchment root (`chmod 0600`, atomic tmp+replace; name regex `[A-Za-z_][A-Za-z0-9_]*`); `set`/`names`/`get`/`remove`. **`names()` returns `[{name, set_at}]` — never values**; `get()` is for internal resolution only (there is **no read-back endpoint**). `create_app` instantiates the store on `app.state.secret_store` and wires `credentials.set_secret_provider(store.get)` (a module-level provider so `${secret:}` resolves deep in a driver). **API** (`/api/secrets`, all `dependencies=[auth.full]`): `GET` (names only), `POST {name,value}` (422 on bad name), `DELETE /{name}`. **CLI**: `duckstring secret set NAME` (value **prompted, hidden — never in argv**), `secret ls`, `secret rm NAME`. **UI**: `SecretsMenu` (the **Secrets** button under Collapse-all, top-right `ControlsPanel`, **full access only**) lists names + a write-only set form; the Spout add form offers a stored-secret datalist of those names. **Encryption-at-rest is deliberately skipped** (plaintext 0600; the design decision is the author's — folks who care use HTTPS + don't transmit). **`set` DOES transmit the value over the wire** (POST body) — by design; use HTTPS. The store is **excluded from the `catchment archive`/download bundle** (`routes/catchment.py` `_SKIP_NAMES`) so secrets never travel in a state bundle. Tests: `tests/test_secrets.py`.

## Alerts (`alerts/`, `catchment/alert_worker.py`, `routes/alerts.py`, `cli/alert.py`; see `plans/alerts.md`)

Failure & freshness **notifications** — the **observability sibling of a Spout** (same shape: operational config not in `pond.toml`, a scheme-selected driver seam, `${env:}`/`${secret:}` creds reused from egress, an async worker that **never cascades a failure into the engine**). A channel is **NOT an engine node** (a notification has no freshness/run semantics) — it is lightweight config + an outbox + a worker; alerting **observes** state transitions the engine already computes and adds no orchestration state. **Built:** webhook + email notifiers, event- and tick-driven emission with root-cause dedup, the outbox worker, CLI + full-gated API. Migration `014_alert.sql` (`alert_channel` + `alert_delivery`).

- **The notifier seam** (`alerts/base.py`): `Notifier` Protocol (`send`/`test`) + `get_notifier(uri)` scheme registry (mirrors `get_egress`); `parse_notifier_destination` validates scheme + `${…}` syntax without resolving. **`WebhookNotifier`** (`http`/`https`) POSTs a body that is both a plain structured event **and** Slack-incoming-webhook-compatible (a top-level `text` summary + the structured fields) — covers Slack, generic receivers, PagerDuty-via-proxy. **`EmailNotifier`** (`mailto:`) uses stdlib `smtplib`; SMTP host/port/user/pass/from from the URI query or `DUCKSTRING_SMTP_*`. An `AlertEvent` (`alerts/event.py`) is the rendered, **sanitised** payload — the error *message*, **never a traceback** (a channel is a third-party surface; same concern as `_redact_tracebacks`).
- **Two firing mechanisms.** *Event-driven* (from `Driver` state transitions): `failure` (a Pond Run gives up / dead-or-silent Duck / Duck-level error), `contract` (the Duck refused to publish — additive-contract break), `spout` (a Spout delivery failed — scoped to its source Pond's name), `recovery` (a failed Pond/Spout clears). *Tick-driven* (in `scheduler_tick` beside `_check_liveness`): `freshness` (a scoped Pond's staleness exceeds the channel's `--stale` bound), + its `recovery`.
- **Root-cause dedup (anti-storm):** failure alerts fire **only for roots** — a blocked-downstream Pond is `is_blocked`, never `is_failed`, and the fail path is only called on the root, so blocked Ponds get no alert; the failure payload carries the currently-blocked names as **blast radius**. Within a root, the outbox `UNIQUE(channel_id, dedup_key)` (`"{kind}:{pond}:{f}"`) is the **fire-once-per-episode** fence (retries at the same failed `f` = one alert; a new failed `f` = a new alert). Recovery is emitted centrally in `_process` (diffing `_alerted_failures`) so **every** clear path (fresher run, wake/force/clear, redeploy) is covered once; a **kill** is intentional, not a recovery. A `recovery` also reaches channels subscribed only to the originating kind (via `match_kinds`), so subscribing to `failure` gets you its recovery without also subscribing to `recovery`.
- **Delivery** (`alert_worker.py`, the egress-worker shape): woken by `Driver._signal_alert` or a 5 s tick, it drains `take_alert_deliveries` → `Notifier.send` in a threadpool (per-send timeout), marking each `sent` or bumping `attempts`/parking `failed` at `MAX_ATTEMPTS` (a dead channel stops retrying but stays auditable). `Driver._emit_alert` is the enqueue seam (matches enabled channels by scope + event filter, `INSERT OR IGNORE`s the dedup-fenced rows) and is **wrapped so a bug in alerting can never break a Pond Run**.
- **Config + surface.** A channel is `(name, destination, scope_name|NULL, events CSV|'all', stale_ms?)` — scoped by pond **name** (survives version/major changes), catchment-wide when NULL; operational config, persisted, survives redeploys. **CLI**: `duckstring alert add --to <uri> [--pond N] [--on failure,…|all] [--stale 1h]`, `alert ls|rm|test|log`. **API** (`/api/alerts`, all `dependencies=[auth.full]` — a destination is an outbound egress surface): `GET`/`POST`, `DELETE /{name}`, `POST /{name}/test` (a connection problem is a 200 `{ok:false,error}`, not a 5xx), `GET /alerts/deliveries` (the audit log). **Deferred:** re-notify cadence (needs an ack surface — that's PagerDuty's job), a `/metrics` Prometheus endpoint, a UI management surface. Tests: `tests/test_alerts.py`.

## Data plane (`dataplane.py`, `iceberg_plane.py`)

The **data plane** is how a Pond *publishes* its tables for, and *reads* them from, other Ponds — the cross-Pond interchange layer, distinct from the DuckDB **registry** where Ripples compute (Ripples always compute on the registry; the data plane is the export/interchange layer only). It is **pluggable** behind `dataplane.get_data_plane()`, selected by `DUCKSTRING_DATA_PLANE` (**default `iceberg`**):
- **`IcebergDataPlane`** (default; **pyiceberg is a core dep — NO SQLAlchemy**): an Apache Iceberg base layer **over Parquet data files** (a metadata/snapshot layer, not a file-format swap). Catalog = our own **`FileCatalog`** (`iceberg_catalog.py`), a `MetastoreCatalog` subclass that stores the `table → metadata.json` pointer in a JSON file — pyiceberg's only embedded catalog is `SqlCatalog`, which pulls SQLAlchemy for exactly that pointer, so we replace just the pointer store and inherit all the Iceberg machinery. One catalog **per `name@major`** at `{data_dir}/catalog.json` (namespace `pond`) — a per-line catalog (deviates from the original plan's single root catalog) to keep major-line isolation physical and avoid cross-Duck contention; single-writer-per-line, so commits need no rowcount/optimistic-concurrency check; saves are atomic (`os.replace`) for cross-Pond readers. Write = pyiceberg Arrow `overwrite`, one commit per Pond Run, snapshot stamped `duckstring.f=<iso>`. Read = DuckDB `iceberg` extension (`iceberg_scan` on the snapshot metadata; `prepare(con)` loads it — a one-time download, so an offline Catchment uses `parquet`). A flat `{table}.parquet` **sidecar** is written alongside each commit so ducts/draws + direct file-serve + the not-yet-Iceberg-Source fallback stay behaviour-neutral. `catchment archive` includes the catalog (`catalog.json`, a plain file) + Iceberg dirs via its root walk.
- **`ParquetDataPlane`** (`DUCKSTRING_DATA_PLANE=parquet`): a **wholesale** plain-overwrite output → one `{table}.parquet` in the line's `data/` dir, overwritten per run (atomic tmp+replace). An **append-only** Trickle table (append history, `__changelog`, warm `__band`, `__droplog`) → a **directory of per-run parts** `{table}/{f}.parquet` (one `_duckstring_f`-homogeneous file per run, warm bands one per fold), so a run writes only its delta (O(change), not O(history)); `_export_parts` reconciles the parts to the registry's freshness set (writes missing, prunes retention-dropped) and writes a 0-row marker part at the run `f` for an empty (bootstrap-only) changelog so it stays readable as an empty delta (not a coverage-miss). A **merge *main*** is tiered: its cold base → freshness-ordered chunks `{table}__base/` (`_publish_tiered_main`/`_export_bands`; see the merge-main paragraph below). `read_select`/`list_tables`/`table_path` are directory-aware (`read_parquet('{table}/*.parquet')`); part naming is canonical-UTC ISO via `trickle.part_name`/`part_f`. The lightest/offline opt-out.

The interface threads a write **`mode`** (`overwrite` now; `append`/`merge` reserved → raise) and a per-run **`f`** stamp, and reserves the **`_duckstring_*`** system-column namespace (rejected at publish). **As-of read by `f`** is wired as the seam (`read_select(..., as_of=)` resolves the snapshot whose stamped `f <= as_of`); Phase-1 default is latest. Routed through it: executor export, local-runner export+seed, `Pond.read_table` foreign reads, `/api/data`. The duct/draw raw-Parquet *transfer* (poller.py/draw.py) rides the flat layer: a wholesale table ships its single file; an append-only table ships only the **parts newer than the consumer's `landed_after`**, which the consumer drops into its own parts directory (no merge — parts are immutable, idempotent by name). **The merge *main* is a log-structured 3-tier LSM** (`plans/trickle-main-incremental.md`, implemented): **L0** the hot `__changelog` (per-run parts; `read_delta`'s window source) → **L1** warm consolidated freshness-range Z-set bands (`{table}__band/`, written by `fold_warm` when the hot tier outgrows `2×` the chunk threshold, always leaving a hot window so caught-up consumers keep a delta — this raises the delta floor) → **L2** the cold clean base (strictly `d=+1`, one row/PK) published as **size-bounded freshness-ordered chunks** `{table}__base/` (DuckDB `FILE_SIZE_BYTES`), rewritten only at a **cold compaction** (`checkpoint`, k=1: when warm ≥ cold, never below `DUCKSTRING_COMPACT_THRESHOLD`, default 256 MiB — or a per-table override recorded at the merge write). Per-run publish is **O(change)**; the O(base) cold rewrite is amortised by k=1. A merge main read **reconstructs** the current state = latest-per-PK over the clean cold base ⊎ (warm ⊎ hot) filtered `> f_base` — the cold base is **anti-joined by the retraction keys, never grouped** (`DataPlane.read_select` over each plane's `_raw_read_select`; the Iceberg plane serves all tiers from the flat layer, never catalog-committed). The **cold base ships only when its `f_base` advanced** past the consumer's `base_after` (the draw gate), so the large base isn't re-sent every draw; warm bands ride the incremental parts transfer and the consumer reclaims tiers subsumed by an advanced `f_base`. **Tests**: `test_runtime`'s broad e2e suite is pinned to `parquet` (fast/offline); the Iceberg e2e is `test_demo_chain_runs_on_iceberg_end_to_end` + `test_iceberg.py`. See `plans/data-plane-iceberg.md`.

## Trickle — incremental I/O via Z-sets / DBSP joins (`duckstring/trickle/` — see `plans/trickle-dbsp.md`, `plans/trickle-dag.md`)

> **Intentionally a self-contained subpackage, ready to extract.** `duckstring/trickle/` (`context.py` / `io.py` / `builder.py` / `agg.py`) is a standalone DBSP-style incremental engine over DuckDB that depends on **nothing else in duckstring** — its only host seam is `context.Context` (a DuckDB connection + a stable **epoch** `f`/`previous_f` + `read_table`/`read_delta`), which `Pond` satisfies structurally. It owns its own `SYSTEM_PREFIX` and `NEVER` (the data plane re-exports `RESERVED_PREFIX` from it, keeping a single source of truth and the dependency pointing duckstring→trickle, never back). This decoupling is deliberate so the engine can be lifted into its own pip package whenever a second consumer or a standalone-library push warrants it; until then it ships in-repo on one release. The legacy top-level modules `trickle_io.py` / `trickle_builder.py` / `agg.py` are thin PEP-562 forwarding shims kept so existing/public imports (incl. `from duckstring import agg`) and all internal call sites are unchanged — on extraction they become the three one-line adapters to repoint. **Keep trickle dependency-free: never `import` from the rest of duckstring inside `trickle/`; inject host concerns through `Context` (or a function arg, as `read_delta(..., dp=...)` does for the data plane).**

A **Trickle** is a Ripple variant for **incremental** work: it preserves history so a consumer composes its output from just the rows that changed in the window `(pond.previous_f, pond.f]`. Every change is a **Z-set** (a relation with an integer weight `_duckstring_d` per row: `+1` present, `-1` retraction); an **update is a `-1` of the old full image + a `+1` of the new**, so deletions/updates carry **full row images**, not key tombstones — which is what lets the builder join on **any** key (DBSP-style) and propagate deletes soundly. The win is incremental **I/O** *and* the composed delta of a join (joins don't recompute the whole output when only some sources change). **A Trickle is NOT a node type** — there is no `@trickle` decorator; "incremental" is a capability any `@ripple` reaches for by writing history-preserving tables (`append_table`/`merge_table`) and reading deltas. The mode is per-table, chosen at the write; the merge **pk is declared at the write call**, not on the node. The I/O lives in **`duckstring/trickle/io.py`** (reached via `pond` methods or the `duckstring.trickle` package; `trickle_io` is a compat shim). (This replaced the earlier `_duckstring_op`/`_duckstring_hash` upsert/delete-tombstone model + `keys_joining`/`affected_groups` key-set helpers + the FK=PK builder constraint — all gone; the feature was unreleased, so the changelog/sidecar format changed outright with no migration. The old design is `plans/trickle.md`.)

- **System columns** (reserved `_duckstring_*`): `_duckstring_f` (freshness, stamped on every history/changelog row, **and on each merge base row** = its last-write freshness — for as-of reads + the data viewer; stripped by `read_table`) and `_duckstring_d` (the Z-set weight on a merge changelog). The merge **main is a log-structured 3-tier LSM**: a clean **cold base** (folded up to `f_base`, chunked, written only at a cold compaction) + **warm Z-set bands** (`__band/`, `fold_warm`) + the per-run **`__changelog`** (hot), with the current state **reconstructed on read** (latest-per-PK over cold ⊎ warm ⊎ hot, filtered `>f_base`; cold anti-joined by the retraction keys, never grouped; see `trickle/io.py:reconstruct_sql`/`_clog_union_sql`). `_duckstring_f` is read as a **content predicate** (`WHERE _duckstring_f > previous_f AND _duckstring_f <= f`), **never a snapshot cursor**; the consumer's window read prunes by content predicate regardless.
- **Write API** (Pond handle): `append_table(name, rel, *, pk=None, fail_on_conflict=True)` — insert-only history (all `+1`; idempotent at `f`); `pk` is optional metadata, and with `pk` set `fail_on_conflict=True` (default) asserts uniqueness across the batch + existing history (raises pre-write), `False` is the trust-the-writer fast path (no check; a no-op when `pk` unset). `merge_table(name, rel, *, pk)` — `pk` **required**; the **complete current state**; diffs it against the **reconstructed prior state strictly before `f`** (`reconstruct_current(con, name, before=f)`) as a full-row Z-set (`new(+1) ⊎ prior(-1)` consolidated) and **appends the diff to the `__changelog`** (no main write). `apply_zset(name, zset, *, pk)` — the low-level primitive; appends a Z-set change to the changelog directly; the builder's incremental path uses it. **pk required at the merge/apply call**. **Changelog-only, atomic apply**: `apply_zset` consolidates once (full-row `GROUP BY`, BIGINT weight) then commits the `__changelog` `DELETE(@f)+INSERT` in one transaction — the main is never touched per run (it is the log-structured base, folded only at a checkpoint). The `before=f` prior + replace-the-`f`-window makes a **same-`f` re-merge** and a **replay** both correct (the diff is against the pre-`f` state, replacing the window with the full net change). An **empty** consolidated change leaves the changelog untouched. `fold_warm(con, name, target_f)` moves the hot changelog slice `(f_warm, target]` into a consolidated warm band (collapsing a→b→c→d, raising the delta floor); `checkpoint(con, name, target_f)` (cold compaction) folds `cold ⊎ warm ⊎ hot≤target` into a fresh clean base, clears the warm tier, and advances `f_base`/`f_warm` (the amortised O(base) write; lock-free — reads are latest-per-PK so a base swap is idempotent under overlap).
- **Read API**: `pond.read_delta("src.table")` → a **`Delta`** (`.zset` — user cols + `_duckstring_d`; `.is_full`; `.upserts`/`.deletes` are derived conveniences) over the window. Append → history window all `+1`; merge → changelog window **consolidated by full row** (multiple updates / delete-then-re-add collapse); overwrite (plain Ripple) → if its published `f` advanced past `previous_f` a **full read** (`is_full=True`), else an **empty** delta (an unchanged Ripple is a free stable operand); bootstrap (`previous_f=NEVER`) / coverage-miss → full read. `read_table` of a Trickle source is the **clean current state** (system columns stripped). The coverage check keys off the published **`floor`** (`_trickle.json` sidecar): valid only when `previous_f >= floor`; set at bootstrap/refresh (`= run.f`, empty changelog) and raised by retention.
- **Builder DSL** (`trickle_builder.py`): `pond.trickle(spine).join(pond.trickle(dim), on=…, how="inner").filter(…).mutate(col=…).select(…).merge(name, pk=…)`. The builder records an **operator DAG** of `_Source`/`_Join` nodes (`plans/trickle-dag.md` — v2: a DAG of binary incremental joins, replacing the old spine+telescoping star). **Any equi-join key** (`on` = shared column name(s), or a `{left: right}` dict; an `alias.col` name disambiguates). **`how`** ∈ inner/left/right/full/semi/anti (`_JOIN_SQL`; `_LEFT_ONLY` = semi/anti) — **all maintained incrementally**, including the outer joins' NULL-padded incomparables; there is **no privileged spine** and no right/full solo-comprehensive restriction. `.join`'s operand may itself be a join DAG, so **bushy `(a⋈b)⋈(c⋈d)` and snowflake shapes** compose directly (the old must-be-bare-dimension guard is lifted); an operand carrying its own `.filter()`/`.mutate()`/`.select()`/`.aggregate()`/`.sql()` raises (attach those to the result). **The maintenance rule (one rule, every `how`):** each binary join node is maintained by the **affected-key recompute** (`_join_delta`): `K = πₖ(δL) ∪ πₖ(δR)` (changed sides' join-key values), then recompute the join restricted to `key ∈ K` over the **new** states (+1) and **old** states (−1) and consolidate — `_restricted_join` pre-filters **both** inputs to `(keys) IN (SELECT k0 … FROM K)` (the key pre-filter, sound for any `how`, so a small change never scans the other side). Old state = `consolidate(current ⊎ −delta)`, reconstructed only for changed sources (`_reconstruct_old`). **Composes to inline SQL** — every node is a unique-named `CREATE TEMP VIEW` (`_view`) the planner inlines; the gate (`plans/trickle-dag.md` §"Gate result") proved per-node *materialisation* loses to inline at every scale, so the **only** persistence fence is an explicit `.merge()` boundary (which also buys cross-run reuse). Internal columns are **leaf-qualified** `"alias.col"`; the **output pipeline** `_pipeline_sql` layers the ordered `filter`/`mutate`/`select` ops (call order, as nested subqueries) over the compiled DAG state — `_qualify` rewrites their `alias.col` refs to `"alias.col"` (a mutated/bare name passes through, so the two namespaces never collide). **`.mutate(**cols)`** adds computed columns (`SELECT * EXCLUDE` replaces a same-named one; siblings in one call see the input, chain calls to build on a new col; a mutated col can be a `pk` but **not a join key** — `on` resolves only against source leaves). **`.select`** is optional — with none the output is the bare `*` (`_star_output`: qualified→bare, equi-join keys deduplicated via the `_join_key_finder` union-find, any other bare-name collision raises). `.filter` after a `.mutate` may reference the mutated column (the pipeline is row-local + deterministic, so it distributes over the Z-set delta — incrementally free, same argument as a computed `.select`). **`.alias(name)`** names a source (`_alias_for` = explicit else positional `s{i}` by left-to-right leaf order; uniqueness validated; the spine-PK fast-path regex keys on the spine's effective alias); `on` resolves per subtree via `_resolve_col` (leftmost-preferred on the left, ambiguity raises). **A builder method exists only if the engine maintains it incrementally; everything else → `.sql()`.** **`.sql(query)`** is the **comprehensive escape hatch** (`.alias()` required to name the table; a joined DAG with no `.select` resolves columns via the bare `*`): collapses the composition to one materialised relation (`_materialised`), runs `query` (a SQL string, or an Ibis expr compiled via `ibis.to_sql(..., dialect="duckdb")` — ibis imported lazily, never a dep), and returns a **comprehensive-mode** builder — `.join`/`.select`/`.mutate`/`.filter` raise after it, but `.merge()`/`.append()` still diff/append-filter the relation so the **output delta stays incremental**. This is the `priced→revenue` boundary collapsed into one ripple (ergonomic, not a compute win — the aggregate still rescans). **`.schema()`** → `{col: duckdb_type}`; **`.to_ibis_schema()`** maps it to ibis type-strings (raises on an unmapped type) for `ibis.table(...)`. **`pk=` on `.merge()` is required** (the output identity — must be genuinely unique; TypeError if omitted, BuildError if empty). A term for an unchanged source drops out (free — its delta is empty, so no `K` term and the restricted join yields nothing). **Comprehensive fallback** — any source `is_full` (bootstrap/coverage-miss/changed-Ripple) or over its threshold `p` reads as `is_full`, propagating up so that subtree (and so the whole output) recomputes via `_full_join` and diffs against the **materialised prior output** (the last-written main, **read not recomputed**) via `pond.merge_table`. `BuildError` for a join operand carrying filter/mutate/select/agg/sql, an empty merge key, an ambiguous join key, a `*` output with an unresolvable (non-join-key) name collision, a `.mutate` using the reserved `_duckstring_` prefix, or an output (`.select`/`.mutate`/`*`) omitting the PK. **Change-fraction threshold** `pond.trickle(ref, p=0.3)` (per source): if the delta touches > `p` of the source's current rows, that source reads `is_full` (comprehensive); `p=1.0` disables the check (and skips the count). **Strategy escapes** on `.merge()`/`.append()` (both default `True`, manual overrides — measure first): `ivm=False` ignores deltas and recomputes the whole output via `_full_join` diffed vs the stored main (also disables the append spine-PK fast path); `key_filter=False` keeps the delta composition but skips the `IN (…)` pre-filter (`_restricted_join` joins full new/old states) — for when the change is large enough to trip `p` anyway. Threaded via `self._key_filter`.
- **`.append()` terminal** (`trickle_builder._compute` is the shared ΔO step behind both `.merge()` and `.append()`): `pond.trickle(...).…​.append(name, *, pk=None, fail_on_conflict=True, log_drops=True)` writes the composed result to an **append** (insert-only history) Trickle instead of a merge main+changelog — for a **monotonic** transform (output rows only added, never updated/retracted; e.g. an append fact stream ⋈ stable/SCD dims). `trickle_io.append_zset` consolidates ΔO then: a **retraction** (`d<0`) or a `+1` whose `pk` is in history with a **different** image is a conflict (an *identical*-image collision is a benign idempotent skip); `fail_on_conflict=True` raises pre-write, `False` drops them (history wins, past frozen) and — with `log_drops` — appends them to a **`{name}__droplog`** companion (user cols + `_duckstring_d` + `_duckstring_f`), an append-only diagnostic **published alongside the table like `__changelog`** (`publish_plan` exports it as a file but gives it no sidecar entry; `schema_contract`/Iceberg treat it like `__changelog`). A `pk` dup **within one run** (distinct images) always raises (no run-level recency to disambiguate). `pk=None`+`False` skips the checks (fast; sound only when dups/past-changes are impossible). A comprehensive recompute is tagged `+1` and append-filtered against history (re-derived identical rows skip, changed ones conflict) — so coverage-misses/replays don't spuriously fail. **Spine-PK fast path** (`_spine_pk_passthrough`): when the output PK is a verbatim `s0.<col>` pass-through of the **spine's** declared PK *and* `fail_on_conflict=False`+`log_drops=False`, a dim delta can't change the result (changed facts are dropped-and-unlogged either way), so the builder skips the ΔO composition entirely and computes `new spine rows (pk ∉ history) ⋈ current dims` (`_full_join(spine_rel=…)`, the spine = the leftmost leaf), turning a big-dim-change-drives-spine-scan into an O(spine delta) lookup. Detection is conservative (only the verbatim `s0.<col>` form; any computed/renamed/non-`s0` PK column falls back to the always-correct general path — a false positive would wrongly drop rows, so the parse must never guess).
- **`.aggregate(by, **metrics)` / `.group_by(by).aggregate(...)`** (`trickle_io.apply_aggregate`; metrics = `duckstring.agg` specs — **count/sum/mean/min/max/var/stddev**, the **weighted** family **weight_total/weighted_sum/weighted_average**, the two-variable **co-moments** **covariance/pearson_correlation/ols_slope/ols_intercept**, the **payload extremes** **argmin/argmax**, the **semigroup reductions** **bool_and/bool_or/bit_and/bit_or**, and **product** (retractable log-sum-exp: count+n_zero+n_neg+Σlog|x| → `(−1)^n_neg·exp(Σlog)`, a float — not bit-exact for big int products); var/stddev/covariance take `how="sample"`(default)/`"pop"`). A grouped **merge Trickle** keyed by `by` (output `pk` defaults to `by`); terminal-bound to `.merge()` (`.append`/further joins after it `BuildError`). Raw accumulators live in a registry-only **`_duckstring_agg_{name}`** companion (`AGG_STATE_PREFIX`, hidden from publish): `_a_cnt`; per additive col `_a_sum/_a_cnt/_a_m2` (the **centred second moment** `M2=Σ(x−x̄)²`, not a raw sum-of-squares); per extreme col `_a_min/_a_max`; per co-moment pair `_c_n/_c_sx/_c_sy/_c_m2x/_c_m2y/_c_cxy`; per weighted unit `_w_num/_w_den`; per argmin/max `_g_key/_g_arg`; per semigroup `_s_val`; per product `_p_cnt/_p_nz/_p_nn/_p_sl`. The published main holds only the derived user columns. **Numerically robust:** var/stddev/covariance/correlation/ols are maintained by the parallel **Chan/Pébay merge-in/merge-out** of centred (co-)moments — never the cancellation-prone `Σx²−(Σx)²/n` (the partition moments are computed about partition means in a **two-pass** `dacc`; comprehensive rebuild uses DuckDB's stable `var_pop`/`regr_*`). See `plans/trickle-agg.md`. **Incremental path** (input ΔO): fold the distributive sums additively + merge each partition's centred (co-)moments in/out per affected group (O(δ)); **min/max/argmin/argmax/bool*/bit* extend in place from the inserts**, but a group with any retraction **rescans** its current membership (the rescan families are `RESCAN_KINDS`; `current` = the builder's `_full_join()`, passed in when `needs_current`, filtered to retracting groups — append-only never rescans). **f-stamped** so a replay skips already-applied groups; emit `new(+1) ⊎ old(−1)` for affected groups → `apply_zset`. **Comprehensive path** (bootstrap/coverage-miss/over-`p`): rebuild the accumulators wholesale (`_agg_rebuild`) + `merge_table`. A group whose count hits 0 is dropped + retracted.
- **`.along(col)` + `.accumulate(by, **metrics)`** (`trickle_io.apply_accumulate`; metrics = `duckstring.acc` specs **sum/count/min/max/first/product/ema/tema** + **scan** (a custom fold `fn(state,row)->(new_state,output)`, state JSON-persisted) — the `acc.` prefix marks them as *accumulated*, so `acc.sum` is a running sum) — **order-DEPENDENT per-row scans** (vs `.aggregate`'s order-independent reductions). `.along(col)` declares the **monotonic order axis** (non-decreasing with freshness — a precondition, not a sort); `.accumulate(...)` is a **non-terminal transform** (output cardinality = input — it enriches each row with its running value in `.along` order within its `by` group) finished by **`.append()`** (append-only, input monotonic in `.along`) **or `.merge()`** (**retraction-aware** — `apply_accumulate_merge`: carries per-group end-state and splits affected groups via `_classify_affected` — a **tail-only** append resumes from carried state and folds only the new rows (`O(new)`), a **past-changed** group is re-folded over its current membership and merge-diffed vs prior (`O(group)`); no monotonic constraint). The order-dependent **custom reduction** is **`agg.reduce(fn, init)`** via `.along().aggregate(by, m=agg.reduce(...)).merge()` (`apply_ordered_reduce` = the same re-fold, collapsed to one value per group; needs `.along`, can't mix with order-independent metrics). Append-mode keeps per-group **carried fold-state** in a registry-only **`_duckstring_acc_{name}`** companion (`ACC_STATE_PREFIX`, hidden): the accumulators + last `.along` value, f-stamped for replay. The scan is a **Python fold** continued from the tail (uniform across all metrics incl. the recursive ema/tema and the FIFO-buffer lag/conv/scan — `O(new rows)`/run); an all-scalar-seed scan (sum/count/min/max/first with a `by`) takes a **SQL-window fast path** (`_accumulate_windowed`: one window pass over the batch + the carried per-group seed, also incremental — `_ACC_WINDOWABLE`). **Bootstrap/coverage-miss** re-folds from scratch (idempotent via `append_zset`'s conflict-skip). A row below its group's `.along` high-water mark, or a retraction reaching the scan, **raises** (the monotonic + append-only contracts). `agg`/`.aggregate` and `acc`/`.accumulate` are the two namespaces/operators. See `plans/trickle-agg.md`.
- **Chained merges/appends** (`.merge()`/`.append()` return a `TrickleBuilder` rooted at the just-written table): `a.join(b).merge("ab",pk=…).join(c).merge("abc",pk=…)` materialises intermediates **mid-chain in one ripple** — same reuse win as splitting into a downstream Trickle (stores `ab`'s trace, so a `c`-only run reuses it instead of recomputing `a⋈b`; also the only way to join on a key that exists only *after* an upstream join), minus the second ripple's boilerplate (and minus the parallelism a separate ripple under a Wave gives — a chain is explicitly sequential). The returned handle is the next **spine**; its in-run delta is read back from the registry via `trickle_io.read_registry_delta` (dispatches append vs merge mode; NOT a name re-read — nothing is published mid-run, so a downstream join can't go through the data-plane `read_delta`; threaded as `_spine_delta`). A composed builder still can't be a *dimension* (the bare-source guard); keep it on the spine.
- **Evaluation order** matters for cost (not correctness — the sum is order-independent): volatile/small sources inner, large/stable outer, so most runs reconstruct no prior state and the surviving term is the small delta against indexed states. Not yet auto-tuned (spine-first declaration order); a large reused intermediate is the signal to **split a Trickle** or **chain through a mid-chain `.merge()`** (every Trickle boundary — including a chained one — materialises a reusable history). **Determinism contract**: retractions cancel by full-row identity, so projections must be deterministic (no `now()`/`random()`/unstable agg).
- **Retention** (`retain_t`/`retain_n`, opt-in): trims old history/changelog rows at write time — a **lag SLA, not correctness** (a consumer behind the window full-reads the clean main). The watermark is `min(_duckstring_f)` over what remains.
- **Following a Ripple**: a changed overwrite Ripple source forces the consumer's comprehensive path (no old state to retract against → diff `O' − main`); an *unchanged* one is a stable history operand. Either way any table is a valid `.trickle()` source — promote a Ripple to a native Trickle upstream to make its changes incremental (a zero-touch change for consumers).
- **Mode/PK/floor + source `f` travel** via the `_trickle.json` **sidecar** (every published base table gets an entry — `{mode,pk,floor,f}` for a Trickle, `{mode:"overwrite",f}` for plain output; the `f` stamp is what lets a consumer detect whether an overwrite source advanced) + a registry meta table `_duckstring_trickle`. `dataplane.publish_plan(con, data_dir, f)` validates non-Trickle tables, exempts Trickle ones, and writes the sidecar.
- **Cross-Catchment incremental draw** (`routes/draw.py?after=&base_after=`, `poller._land_transfer`): the consumer sends `after = landed_after(...)` (the max part `f` it already holds, read from part filenames); the producer ships only the append-only **parts** with `part_f > after` (append history / `__changelog` / warm `__band` / `__droplog`, zipped under `{table}/{f}.parquet`) and the consumer drops each into its own parts dir (idempotent by name — no merge). A merge main's **cold base** is wholesale but ships **only when its `f_base` advanced** past the consumer's `base_after` (so the large base isn't re-sent each draw); the consumer wholesale-replaces the base dir on a cold compaction and reclaims warm/changelog parts `≤ f_base` (subsumed — reconstruct filters them, so this is storage-only). The sidecar always travels.
- **Iceberg is overwrite-only** (`iceberg_plane.export`): only plain-Ripple **overwrite** output is committed to Iceberg (one `f`-stamped snapshot per run, bounded by `_prune` to the latest N). **Append-only Trickle tables** (an append history, a merge `__changelog`, an `__droplog`) and a merge **main** base are **not** committed — an append-only table's current snapshot would reference every data file ever appended, so the Iceberg metadata grows **O(runs)** per run for no read benefit over the flat parts (which prune just as well by Parquet file stats). They are served from the flat per-run parts layer the sidecar export already writes (`IcebergDataPlane._raw_read_select` falls back to `ParquetDataPlane._raw_read_select` whenever a table isn't in the catalog), so their publish is **O(change)/run**. The flat parts read honours the **as-of** seam via a row-level `_duckstring_f <= as_of` predicate (whole f-homogeneous parts are stat-pruned). Keep: `_duckstring_f` exported under `SET TimeZone='UTC'` (pyiceberg rejects non-UTC tz). (This replaced the earlier per-run Iceberg `_append_commit`/`_sync_retention`/`duckstring.last_f`-cursor design, which was O(runs) on every publish.)
- **A returned relation must not depend on a shared-name transient view** (a later call re-creates it → the lazy relation re-binds): `read_delta` inlines its window consolidation as a self-contained subquery; the builder names its per-source views with `trickle_io.unique_name(...)` — preserve that.
- **Demo**: `duckstring pond demo --trickle` scaffolds `orders` (append) → `catalog` (merge, CDC on price drift) → `priced` (the builder) → `revenue` (comprehensive aggregate); `--ripple` is the overwrite set. `priced`/`revenue` ship Trickle-shaped puddles.
- **DBSP maturity / deferred** (see `plans/trickle-dbsp.md`, `plans/trickle-dag.md`): this is the DBSP **incremental-join + distributive-aggregation core** done correctly over an acyclic query class with SQL recomputation (micro-batch-at-`F`, composed to inline SQL — not a maintained indexed dataflow). Done: a DAG of binary incremental joins (all six `how`, bushy/snowflake); incremental aggregation for the distributive/algebraic set (count/sum/mean/min/max/var/stddev) + the weighted family + two-variable co-moments (covariance/correlation/ols) — all retractable and numerically robust (`plans/trickle-agg.md`). Missing vs a full engine, in priority order: order-dependent scans done (`acc.*` / `.along().accumulate()`, `.append()`-only); **skewness/product** (deferred — M3 merge-out and running products need the rescan / log-sum-exp treatment to stay safe), **holistic aggregates** (median/percentile) and **DISTINCT** (need different retraction-aware operators — they stay a downstream `.sql()` comprehensive step), **persistent indexed traces** (cross-run; per-run intermediates are inline, not materialised — the `.merge()` boundary is the only persisted trace), recursion/fixed-point, automatic incrementalisation. Also still deferred: `pond.state_dir` for an external stateful-IVM engine.

## Version contract (`schema_contract.py`; Phase 2 — implemented)

Two parts enforce that a major line stays compatible so a Sink pinning `name@major` is safe:
- **`min_version`** (`pond_to_pond.min_version`) — enforced **at deploy** (`routes/deploy.py`): a Sink under-pinning its Source, or a Source selection that regresses below an existing downstream pin, is rejected (422). A major bump is the escape hatch.
- **Schema additive-check** — enforced **at publish** (a Pond's output schema is a runtime result, so it can't be vetted earlier). Captured per `pond_version` on each accepted run (`pond_version_schema`, migration `005`; reported by the Duck on `run_completed`, stored by `Driver._capture_schema`). The Catchment ships the major line's **high-water schema** as the `begin_run` job's `contract` (`Driver._contract_for`) — `None` for a first run or a **rollback** (selection ≤ a previously-accepted version → governed by `min_version`, the gate is **forward-only**). The Duck vets its output **before publishing** (`duck/executor._export_data` → `contract_violations`): a violation raises `ContractViolation`, **aborts the publish (live tables keep last-good)**, and reports `contract_failed`; the Catchment fails the Source at that Run and blocks downstream. A contract violation is **just a Pond failure with a contract message** — it reuses `fail_pond`/`derive_blocked`, no new engine state. Breaking changes go via a **major bump** (old line keeps running; Sinks re-pin on their own schedule). Additive output (new tables/columns) is always fine; drops/removed-tables/type-changes are violations. Deferred: pinned-minor precision (so an unused mid-major column can be dropped on a minor) + a dedicated `contract` blocked sub-reason in the UI.

## Fault tolerance (implemented — `theory.md` "Fault Tolerance" is authoritative)

Two retry budgets (default 0), live on the Pond and editable via `control failure-budget` (`pond_retry` table; seeded on deploy from the `pond.toml` / `pond_version` defaults, then operator-owned):
- **immediate_retries** — Ripple-Run retries *within* one Pond Run (per-frontier, consumed by the Duck via `worker.immediate_left`).
- **source_retries** (on-change) — whole Pond Runs the Catchment re-attempts when a Source updates.

Pond fault state on `PondState`: `is_failed` (a Run gave up, not yet superseded), `failed_f` (freshest failed Run — the recovery watermark; the run-gate keys on `start_f`, this is for clearing/telemetry), `failures` (count vs `source_retries`), `is_blocked` (a required Source is failed/killed/blocked — **derived and propagated** downstream via `derive_blocked`), `is_killed` (operator Kill; terminal). Gating: a failed Pond only re-runs via the on-change path (`sourceF > startF`, while `failures <= source_retries`); a Run completing fresher than `failed_f` clears the episode; a **blocked-but-not-failed** Pond still drains existing Source output but never solicits; a killed Pond is fully gated.

Failure sources (every type produces a **message**; ripple/Duck exceptions also a **traceback** — surfaced in Run Detail):
- **Ripple error** → Duck spends `immediate_retries` (per-frontier), then reports `failed(F = ripple.startF, error, traceback)`; the Catchment fails the Pond at that Run, counts it, blocks downstream (`engine.fail_ripple`).
- **Duck-level error** (e.g. ledger write) → the Duck reports `pond_failed` (attributed to its last `begin_run`) and exits.
- **Dead / silent Duck** → `Driver._check_liveness` (in `scheduler_tick`, only for `SubprocessLauncher`) fails an in-flight Pond whose process is gone (`proc.poll()`), or whose last contact aged past 60 s (`engine.fail_pond` at `start_f`).
- **Stuck Run** → the Duck's watchdog reports `pond_failed` if it has outstanding work but no Ripple running for 30 s.
- **Kill** → `Driver.kill` terminates the Duck and parks the Pond `killed`.

Recovery: a failed Pond with on-change budget re-runs on the next Source change (respawning a Duck); redeploying a fixed artifact auto-clears the failure (`Driver.clear_on_redeploy`); `control clear` / `force` / `wake` clear it manually. Run history records **one row per attempt** (`ripple_run` keyed on `retry`) with `error` + `traceback`. `_check_liveness` skips failed/killed/blocked Ponds; `clear` rolls `start_f → end_f` so the abandoned Run isn't re-failed.

Concurrency hardening: SQLite connects with `PRAGMA busy_timeout`; DuckDB writes + the Parquet export retry transient locks via `core.retry_on_lock` (export uses a read-write connection, never `read_only`, to avoid clashing with pipelined Ripple connections).

Duck/restart resilience (pre-existing): in-flight Runs complete without the Catchment (events buffer + replay idempotently on `F`); on (re)start the Duck reconciles against its ledger and re-runs **only incomplete Ripples**. `Driver.reload` rebuilds engine state from SQLite (demand/freshness from `pond_state`/`pond_target` incl. the fault fields + `pond_retry`, `gen` from `pond_run` counts, per-Ripple `end_f` from successful `ripple_run` rows); `resume_incomplete` re-dispatches `pond_run` left `status='running'`.

## Catchment database

SQLite `duck.db` at the catchment root. Schema in `catchment/schema/001_init.sql`, applied by `catchment/db.py:migrate()`. New migrations are numbered SQL files (`002_*.sql`, …). All file paths stored in the DB are **relative to the catchment root**. Freshness/targets are stored as UTC ISO-8601 text.

### Schema (identity is a three-table split)

- **`pond_name`** — the abstract named entity (`name`, `kind` ∈ inlet/pond/outlet, `git_branch`).
- **`pond_version`** — a specific deployed snapshot (`pond_name_id`, `version`, `major`, `source_path`, retry config). Immutable artifact; topology + run history key off this.
- **`pond`** — the **selected** version, one per `(pond_name, major)` → `pond_version` (upserted on deploy). This is "the Pond" and the FK target for all live demand/freshness/graph tables.
- Topology (keyed on `pond_version`): `ripple`, `ripple_to_ripple` (intra-pond edges, all required).
- Contract (keyed on `pond_version`): `pond_version_schema` (one row per output `(table, column, type)`, captured on accepted runs; reserved `primary_key` flag for Trickle) — see Version contract.
- Live state (keyed on `pond`): `pond_to_pond` (sink `pond_id` → source `pond_name_id` + `source_major`, so a sink can deploy before its source), `pond_state` (start_f/end_f/d_ms/has_pull/has_received_pull **+ is_failed/is_blocked/failed_f/failures/is_killed/pull_local**), `pond_target` (push target set), `pond_retry` (immediate_retries/source_retries — live budgets), `pond_window` (PK `(pond_id, name)`), `pond_trigger` (PK `pond_id`; kind wave/tide, bound_ms), `pond_spout` (PK `(pond_id, name)`; egress bindings — table/destination/mode/schedule + the worker's watermark/retries/failures/is_failed/error, see Egress).
- History (keyed on `pond_version` + freshness `f`): `pond_run` (`status` ∈ running/success/failed/killed, `error`, `traceback`) and `ripple_run` (**PK includes `retry`** — one row per attempt = the retry trace; `status`, `error`, `traceback`). `started_at`/`finished_at` are the Duck's wall-clock execution span (reported on the `ripple` event) — the UI's run durations. All timestamps are UTC ISO-8601 (tz-aware).
- The **per-Pond run ledger is NOT in `duck.db`** — it lives at `ponds/{name}/m{major}/pond.db` (owned by `engine/pond.py`): the Duck's operational/recovery record (`ripple_run_state`, `pond_run`). The Catchment's `pond_run`/`ripple_run` are the canonical history.

## Orchestration model (theory.md is authoritative)

Freshness-based Kanban. The Pond is a packaging/versioning boundary; its `start`/`end` are zero-duration boundary nodes carried as Pond state, so Pond and Ripple share the same rules.

- **Freshness `F`** — a UTC timestamp per node: the run-start time of the oldest root feeding it (with windows, the "fresh until" window end). `NEVER` (`datetime.min`) is the sentinel for never-run. Staleness = `now + D - F`.
- **Pull** (Tap/Wave) — a `hasPull` token; a node runs when a parent is fresher (`sourceF > startF`) and re-arms parents on start. Cold-start guards use **startF** (`source.startF <= this.startF`).
- **Push** (Pulse/Tide) — a **set** of unsatisfied target freshnesses; run when `sourceF >= min(targets)`, clearing every target reached. Pond run start stamps every Ripple with `Pond.startF`. (The control verbs map onto these: **wake** = a non-propagating one-shot pull, **force** = a same-freshness recompute, **sleep** = clear demand, **kill** = terminate.)
- **No-change skip** (`changed_f`, plans/no-change-skip.md — built) — a **second** freshness stamp `PondState.changed_f` (≤ `end_f`): when this Pond's OUTPUT last actually changed. `start_f`/`end_f` advance on *every* run (the heartbeat); a **pass** advances them but **holds `changed_f`**. In `start_pond_run`, a Pond with Sources is completed in-engine as a **pass** (no `BeginRun`, no Duck — drained via `drain_passes` → `_record_pass` writes a no-change `pond_run`) when `max(Source.changed_f) <= prior_f` (strict, vs the **prior** run freshness — *not* the new `start_f`); `must_run` (Inlet/`always_run`/force/refresh/repair/draw/spout) bypasses it. The Duck reports `changed:bool` on the run-completing `ripple` event + `run_completed` (held → `complete_ripple(changed=)`); `pond.skip()` (per-`f` skip sink) and `pond.sources_changed()` (engine verdict, threaded `BeginRun.sources_changed` → job → `DuckCore`) back the side-effect pattern; `@ripple(always_run=True)` ORs up to `Pond.always_run`. The Trickle no-change signal is **user-controlled** (like overwrite, not automatic): `merge_table`/`append_table`/`apply_zset` **return a `bool`** (changed?) and the builder terminal exposes **`.was_changed()`** — the user ORs them and calls `pond.skip()` (sound per-call; no cross-ripple aggregation). Migration `013`.
- The four hard-won landmines (startF cold-start guards, push target *set*, Tide `max(targets) ?? startF` clock ref, the run-start ripple stamp) are encoded in `engine/` and guarded by `tests/test_engine.py` — preserve them.

## Conventions

- Table names: **singular** (`pond`, not `ponds`).
- Association tables: **`{parent}_to_{child}`** for many-to-many; `{child}_in_{parent}` for nesting.
- FK columns: **`{table}_id`**; qualify (`sink_id`, `source_id`) when two FKs reference the same table.
- Inter-pond and intra-pond concerns stay in separate tables — do not unify them.
- Freshness/demand state is keyed on `pond` (the selected version); topology and run history on `pond_version`.
- **`pond.f`** exposes the run's freshness to ripple code (Duck passes each ripple's `start_f` through the executor; the local runner stamps one `now()` per run) — the replay-stable watermark/provenance stamp (crash replay + immediate retries re-run at the same F). Documented in python-api.md + the Incremental Ripples guide; first brick of Trickle.
- **No DuckDB replacement scans in Ripple/demo/docs code** (referencing a Python local as a SQL table name, `FROM raw`): it resolves by scanning Python frames and is flaky under the Duck's threaded executor ("don't know what type:" failures on CI). `Pond.read_table` registers foreign Source tables as temp views named after the table — SQL references the table name directly; own tables are queried directly; compose relations via the relation API (`.union`, …) otherwise.
